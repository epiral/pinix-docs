Pinix turns any device into an AI-accessible capability source through two concepts: **Providers** and **Edge Clips**.

## What is a Provider?

A Provider is a connection — a service that registers Clips with a Hub and forwards invocations to them. The Provider protocol (`ProviderStream`) handles:

- **Registration**: telling the Hub which Clips are available
- **Invocation forwarding**: delivering invoke requests to the right Clip
- **Heartbeat**: keeping the connection alive

There are two kinds of Providers:

| Provider type | What it manages | Example |
|---|---|---|
| **Runtime** | SDK Clips (Bun/TS processes) | The built-in Runtime in `pinixd` |
| **Edge Clip Provider** | Self — the Provider *is* the Clip | bb-browser, clip-dock-macos |

The Runtime is a special Provider that manages multiple SDK Clips. An Edge Clip Provider manages only itself.

## What is an Edge Clip?

An **Edge Clip** binds directly to hardware or OS APIs. It implements the Provider protocol itself and connects to the Hub as an independent service.

```
┌────────────────────────────┐
│      bb-browser            │
│  (Edge Clip Provider)      │
│                            │
│  ├─ Controls Chrome        │
│  ├─ Site adapters          │     ProviderStream
│  │  ├─ twitter             │ ◄──────────────────► Hub
│  │  ├─ hackernews          │
│  │  ├─ google              │
│  │  └─ 100+ more           │
│  └─ Browser automation     │
└────────────────────────────┘
```

### Why Edge Clips?

Some capabilities can't be packaged as a TypeScript module:

- **Browser automation** requires controlling a real Chrome instance
- **macOS integration** requires native APIs (screenshots, clipboard, notifications)
- **IoT devices** require direct hardware access
- **Mobile apps** require platform-specific frameworks

Edge Clips solve this by running natively on the device and exposing structured commands through the Provider protocol. To the Hub (and the Agent), they look exactly like any other Clip.

## BB-Browser: The reference Edge Clip

[BB-Browser](https://github.com/epiral/bb-browser) is Pinix's browser automation Edge Clip. It turns your browser into an API:

- **36 platforms**: Twitter, Google, HackerNews, Reddit, Wikipedia, Xiaohongshu, and more
- **103 commands**: search, read, post, monitor, extract
- **Site adapters**: structured extraction from each platform, not generic scraping

BB-Browser is automatically started by `pinix start`. It launches a headed Chrome instance and registers its platform commands with the Hub.

<Aside>
  Chrome runs in **headed mode** (not headless) to avoid anti-automation detection. On macOS it renders offscreen; on Linux, pinixd automatically starts Xvfb.
</Aside>

## clip-dock: Device capability packs

`clip-dock` is the concept of a device-side product that exposes all of a device's capabilities as Edge Clips.

Install `clip-dock-macos` on a Mac, and the Agent can immediately:
- Take screenshots
- Read and write the clipboard
- Send notifications
- Access the file system
- Control system settings

The goal: **one install = all device capabilities available to any Agent**.

## The device abstraction pattern

Edge Clips expose raw device capabilities. Regular SDK Clips can then build abstractions on top:

```
Raw capability (Edge Clip):
  browser.navigate → opens a URL in Chrome

Abstraction (SDK Clip):
  twitter.search → uses browser.navigate + site adapter
                    to search Twitter and return structured results
```

This layered approach means:
1. Edge Clip developers focus on reliable device access
2. SDK Clip developers focus on domain logic
3. Agents get clean, high-level commands
