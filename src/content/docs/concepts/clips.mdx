A **Clip** is Pinix's core abstraction — a self-contained unit that bundles *what to do*, *how to do it*, and *how to compose it* into a single installable package.

Your phone runs on apps. Your Agent runs on Clips.

## The three-layer model

Every Clip is a three-in-one:

### Knowledge Layer

The methodology — *what* the Clip can do and *when* to use it.

A Clip's **manifest** tells the Agent:
- **What I can do** — commands and patterns
- **When you should use me** — matching patterns and entities
- **How to call me** — input schema with types and validation

The Builder's domain expertise is encoded in the Knowledge Layer. This lowers the bar for the Agent model — it doesn't need to figure out *how* to browse Twitter, it just needs to recognize that the user wants to search Twitter and call the right command.

### Capability Layer

The execution logic — browser automation, device APIs, business workflows.

Complex logic runs inside the Clip, not in the model's reasoning. The Agent doesn't need to know *how* to do things, only *who* to call.

### Asset Layer

Composable, reusable, accumulable — Clips use Clips, and the network grows.

A Twitter Clip can depend on a Browser Clip. A Research Clip can orchestrate Search + Twitter + Document Clips. The more Clips you install, the more combinations become possible.

<Aside>
  **More Clips, More Intelligence.** Clips don't add linearly — they multiply. Five Clips can produce dozens of distinct workflows depending on the user's intent.
</Aside>

## Three types of Clips

| Type | Managed by | Language | Install method | Examples |
|------|-----------|----------|---------------|----------|
| **SDK Clip** | Runtime | TypeScript | `pinix hub add` | @pinix/todo, @pinix/twitter |
| **Edge Clip** | Self (Provider) | Any | Provider protocol | bb-browser, clip-dock-macos |
| **Builtin Clip** | pinixd (in-process) | Go | Built into daemon | agent, scheduler |

### SDK Clips

The most common type. Written in TypeScript using `@pinixai/core`, installed from the Registry, managed by the Runtime. When you `pinix hub add @pinix/todo`, the Runtime downloads the package, starts a Bun process, and registers it with the Hub.

### Edge Clips

Bind directly to hardware or OS APIs. A browser Clip controls Chrome. A macOS dock Clip exposes screenshots, clipboard, notifications. Edge Clips implement the [Provider protocol](/edge-clips/provider-protocol/) themselves and connect to the Hub as independent services.

### Builtin Clips

Run inside the `pinixd` process itself. Currently two exist:
- **agent** — the Go Agent Runtime (LLM calls, tool loop, context management)
- **scheduler** — cron-based task scheduling

## Clip lifecycle

```
Install ──► Register with Hub ──► Available for invocation
                                        │
                              Agent/User calls a command
                                        │
                              Clip executes, returns result
```

1. **Install**: `pinix hub add @pinix/todo` downloads from Registry
2. **Register**: Runtime starts the Clip process and registers its commands with Hub
3. **Invoke**: Agent or user calls `pinix invoke todo list`
4. **Execute**: Hub routes to the Clip, Clip runs logic, returns structured result

## Example: How a Clip works

When a user says *"search Twitter for AI news"*, here's what happens:

```
User: "search Twitter for AI news"
  │
Agent reads manifests of all installed Clips
  │
Agent matches: twitter clip has `search` command
  │
Agent constructs: { command: "search", args: { query: "AI news" } }
  │
Hub routes to twitter clip
  │
twitter clip:
  ├── Opens browser (via bb-browser dependency)
  ├── Navigates to Twitter search
  ├── Extracts results
  └── Returns structured data
  │
Agent formats and presents results to user
```

The Agent never touched a browser. It never parsed HTML. It just knew *who to call* and *what to ask for*.
