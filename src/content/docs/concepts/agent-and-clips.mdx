Pinix's Agent is designed around one principle: **Agent thin, Clip thick.**

The Agent's job is to understand what the user wants and route to the right Clip. The Clip's job is to actually do the work. This separation is what makes Pinix Agents work with mid-tier models — you don't need GPT-4 or Claude Opus when the heavy lifting is in the Clips.

## What the Agent does

```
Agent's responsibilities:
  ├── Intent recognition     → understand what the user wants
  ├── Clip selection         → pick the right Clip from its manifest
  ├── Dispatch               → call the right command with right params
  ├── Memory                 → cross-session context
  └── Skills = Clips         → no built-in capabilities needed

What the Agent does NOT do:
  ├── Complex business logic → that's in the Clip
  ├── Device operations      → that's in the Edge Clip
  └── Require SOTA models    → Sonnet-class is sufficient
```

## Why mid-tier models are enough

The Clip's **Knowledge Layer** (manifest) already tells the Agent everything it needs to know:

- *What I can do* — commands, patterns
- *When you should use me* — matching patterns, entities
- *How to call me* — input schema

The Agent only needs to:
1. Read the manifest
2. Match the user's intent to a pattern
3. Construct the correct parameters
4. Call the command

No complex reasoning. No multi-step planning about *how* to browse the web or *how* to send a message. The Clip handles all of that.

<Aside type="tip">
  **Builder's wisdom is encoded in the Clip's Knowledge Layer.** This shifts intelligence from the model to the ecosystem. A well-built Clip makes any Agent smarter.
</Aside>

## Workflow example

User says: *"Find what's trending on HackerNews about AI and save it to a doc."*

```
Step 1: Agent reads manifests
  → hackernews clip: has "top" command, pattern matches "trending"
  → lark-doc clip: has "create" command, pattern matches "save to doc"

Step 2: Agent plans
  → First: hackernews.top → get trending posts
  → Filter: posts about AI
  → Then: lark-doc.create → save filtered results

Step 3: Execution
  → invoke hackernews top → returns structured post list
  → Agent filters for AI-related posts
  → invoke lark-doc create → creates document with results
  → Returns summary to user

What happened inside the Clips:
  hackernews clip:
    ├── Opened browser → HN front page
    ├── Extracted post data (title, score, comments, URL)
    └── Returned structured JSON

  lark-doc clip:
    ├── Authenticated with Lark API
    ├── Created new document
    ├── Formatted content as rich text
    └── Returned document URL
```

The Agent never touched a browser. Never called the Lark API. Never parsed HTML. It made two function calls and got structured results back.

## The multiplication effect

Clips don't add — they multiply.

With 5 Clips installed (HackerNews, Google, Twitter, Lark Doc, Scheduler), the same Agent can handle:

| Scenario | Clips used |
|----------|-----------|
| Tech discovery | HackerNews + Google + Twitter + Lark Doc |
| Competitor research | Google + Twitter + Browser + Lark Doc |
| Daily briefing | HackerNews + Google + Twitter + Scheduler |
| Social monitoring | Twitter + Google + Lark Doc |
| Content research | Google + HackerNews + Lark Doc |

Same 5 Clips, 5+ distinct workflows. No preset workflows, no hard-coded chains — the Agent decides which Clips to use based on the user's intent.

## Builder vs User

| | Builder | User |
|---|---|---|
| Who | Developers with domain expertise | Anyone |
| Does what | Creates Clips | Uses Agent + Clips |
| Needs | SDK, tooling, distribution, recognition | Things done, simple install, no tech knowledge |

### The value chain

```
Builder's expertise → encoded in Clip → available to User via Agent
```

A Builder who understands Twitter's search API wraps it in a Clip. Now any user, through any Agent, can search Twitter without knowing anything about APIs, rate limits, or browser automation.

**More Clips, More Intelligence.**
