Pinix Clip、MCP Tool 和 OpenAI GPTs 都旨在扩展 AI Agent 的能力。它们做出了不同的权衡。

## 对比表

| | Clip | MCP Tool | GPTs |
|---|---|---|---|
| **知识层（Knowledge Layer）** | Manifest（模式、实体、schema） | 工具描述（纯文本） | System prompt |
| **能力层（Capability Layer）** | 完整执行逻辑 | Function call | 有限（Actions） |
| **可组合性** | Clip 依赖 Clip | 不支持 | 不支持 |
| **Token 效率** | 复杂逻辑在 Clip 中运行 | 取决于实现 | 全部在模型推理中 |
| **分发** | Registry + `pinix hub add` | 没有标准分发 | GPT Store |
| **设备能力** | Edge Clip 绑定硬件 | 需要额外实现 | 不支持 |
| **发现** | 结构化 manifest，自动发现 | 来自 server 的工具列表 | 用户选择 GPT |
| **跨工具调用** | Clip 通过 Hub 调用 Clip | 未标准化 | 不可能 |
| **无状态调用** | 是（每次调用自包含） | Server 管理状态 | 基于会话 |
| **多协议** | CLI / MCP / HTTP / Connect-RPC | 仅 MCP | 仅 HTTP Actions |

## 关键差异

### 1. 结构化知识 vs 纯文本描述

MCP Tool 用一段文本字符串描述自己。Agent 必须只根据这段描述推断何时以及如何使用它们。

Clip 提供带类型 schema、模式匹配提示和实体定义的**结构化 manifest**。Agent 不需要猜 —— manifest 是机器可读的。

```json
// MCP Tool: plain text
{
  "name": "search_twitter",
  "description": "Search Twitter for tweets matching a query"
}

// Clip manifest: structured
{
  "commands": [{
    "name": "search",
    "description": "Search tweets",
    "patterns": ["search twitter", "find tweets about"],
    "input": {
      "query": { "type": "string", "required": true },
      "sort": { "type": "string", "enum": ["recent", "hot", "relevant"] }
    }
  }]
}
```

### 2. 可组合性

MCP Tool 是隔离的 —— 一个工具不能调用另一个工具。如果你需要一个组合 search + browser + document creation 的工作流，Agent 必须编排每一步。

Clip 可以**依赖其他 Clip**。Twitter Clip 声明对 Browser Clip 的依赖。当 Twitter Clip 需要打开浏览器时，它会通过 Hub 调用 Browser Clip —— 不需要 Agent 参与。

```
MCP: Agent orchestrates everything
  Agent → Tool A → Agent → Tool B → Agent → Tool C

Clip: Clips compose internally
  Agent → Clip A → (internally calls Clip B) → result
```

这意味着更少的往返、更低的 token 使用量，并且 Agent 不需要理解中间步骤。

### 3. Token 效率

使用 MCP 时，复杂操作通常需要 Agent 和工具之间多轮来回交互。每一次往返都会消耗 token。

使用 Clip 时，复杂逻辑在 Clip 内部运行。Agent 发起一次调用；Clip 在内部处理复杂性。Agent 只看到最终结果。

### 4. 分发与发现

MCP 没有标准包 Registry。每个 MCP server 都要在 `claude_desktop_config.json` 或等价配置中手动配置。

Clip 有 **Registry**（`pinix hub add @scope/name`）、**Marketplace**（从 Web 浏览和安装）以及**共享**（将你正在运行的 Clip 暴露给其他用户）。发现能力内置于平台。

### 5. 设备能力

MCP Tool 没有访问设备硬件的标准方式。你需要为每种设备能力构建自定义 MCP server。

Pinix 有 **Edge Clip** —— 这是设备绑定能力的一等概念。BB-Browser 将 Chrome 变成 100+ 个命令。clip-dock-macos 暴露截图、剪贴板和通知。Provider 协议让任何设备能力都可安装。

## 什么时候用什么

| Use case | Best fit |
|----------|---------|
| Claude Desktop 的快速工具 | MCP Tool |
| 单函数 API 包装 | MCP Tool |
| 包含多个工具的复杂工作流 | Clip |
| 设备/硬件集成 | Edge Clip |
| 可供任何 Agent 复用的能力 | Clip |
| Marketplace 分发 | Clip |

<Aside>
  Pinix 也**支持 MCP** —— 你可以将所有 Clip 作为 MCP Tool 暴露给 Claude Desktop、Cursor 或任何 MCP client。请参阅 [MCP 集成](/zh/getting-started/mcp-integration/) 完成设置。
</Aside>

## Pinix 是互补，不是竞争

Pinix 不替代 MCP。它基于同一个理念（给 Agent 工具），并增加了更多层：

- **结构化知识**，用于更好的工具选择
- **可组合性**，用于复杂工作流
- **分发**，用于生态增长
- **设备抽象**，用于真实世界能力

如果你已经在使用 MCP Tool，可以通过 Pinix 的 Hub 暴露它们，并获得 Clip 生态的优势 —— 路由、发现、共享和跨 Clip 组合。
